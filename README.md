# 车型图像识别系统

一个基于 FastAPI + PyTorch + 原生前端实现的课程项目，用于上传车辆图片，并由 3 个不同架构的模型并行识别车型。系统支持前端结果对比展示、多模型共识判断、用户反馈采集，以及管理员端按模型查看表现。

这个 README 主要是写给项目组成员看的，目的是让队友能够快速了解项目结构、功能亮点、演示方式，并直接作为课程报告和 PPT 的参考材料。

## 项目背景

随着图像识别技术的发展，车辆品牌与车型识别在智慧交通、停车场管理、车辆统计、汽车资讯等场景中都有实际应用价值。相比只做单模型分类，本项目尝试使用多个不同架构的模型共同参与识别，以便：

- 对比不同模型的识别效果
- 分析不同模型在精度和速度上的差异
- 通过投票共识提高最终结果的稳定性
- 为后续模型优化提供反馈闭环

## 项目目标

本项目的目标可以概括为 4 点：

1. 构建一个可以本地运行的车型图像识别系统
2. 使用 3 个不同模型对同一张图片并行识别
3. 提供用户反馈和管理员分析能力
4. 为课程报告、系统展示和答辩准备一个完整原型

## 项目简介

本项目面向“车型识别”场景，核心目标包括：

- 使用多个深度学习模型对同一张车辆图片并行推理
- 对比不同模型的预测结果、置信度和延迟
- 通过多数投票生成共识结果
- 收集用户反馈，支持后续数据修正与模型优化
- 在管理员端分别查看各个模型的反馈表现

## 功能特性

- 三模型并行识别
  - EfficientNet-B3
  - ResNet50
  - MobileNetV3-Large
- 前端上传图片并展示：
  - 多模型共识结果
  - 单模型预测结果
  - Top-3 预测
  - 推理耗时
- 用户反馈采集：
  - 正确 / 有误反馈
  - 正确车型补充
  - 备注说明
- 管理员端看板：
  - 反馈总量统计
  - 各模型单独统计
  - 分模型反馈准确率
  - 平均置信度
  - 最近错误样本
  - 按模型筛选反馈明细

## 系统演示流程

如果队友需要做汇报或录制演示视频，可以按照下面这条顺序来讲：

## 系统架构说明

系统整体采用“前端页面 + FastAPI 后端 + 多模型推理”的结构：

1. 用户在前端上传图片
2. 前端将图片转为 base64 并发送到后端接口
3. 后端统一解码图片，并交给 3 个模型分别推理
4. 后端汇总各模型结果，计算共识结果
5. 前端展示单模型结果和共识结果
6. 用户提交反馈后，后端将反馈记录写入本地文件
7. 管理员端读取反馈并做分模型统计

这套结构的优点是实现清晰、便于本地演示，也容易在答辩时说明模块关系。

## 模型说明

项目中使用 3 个不同架构的车型识别模型进行对比：

1. EfficientNet-B3：兼顾精度与效率
2. ResNet50：经典残差网络，稳定性较好
3. MobileNetV3-Large：轻量级模型，便于后续部署优化

系统会同时调用多个模型，对结果进行展示，并通过共识投票得到最终推荐结果。

## 三个模型的设计思路

从课程项目角度，可以这样概括 3 个模型的意义：

1. EfficientNet-B3  
   兼顾精度与效率，适合作为整体表现较均衡的模型

2. ResNet50  
   属于经典 CNN 架构，便于说明残差网络思想，适合写进报告理论部分

3. MobileNetV3-Large  
   参数量更小，更能体现“轻量化部署”的方向，适合在 PPT 中突出模型对比

这样写报告时，不只是“用了 3 个模型”，而是能说明“为什么要选这 3 个模型”。

## 关于数据集与权重文件

由于数据集、训练日志、缓存文件和模型权重体积较大，且部分权重文件超过 GitHub 普通仓库单文件限制，本仓库默认**不包含**以下大文件内容：

- 数据集图片
- 训练生成的模型权重 `.pt`
- 本地缓存与日志
- 用户反馈数据

如果你需要完整运行项目，请在本地自行准备并放置：

- `backend/data/`
- `backend/data_smoke/`
- `backend/weights/`
- `backend/weights_smoke/`

并确保 `backend/config.py` 中的模型路径配置正确。

## 管理员端说明

管理员端用于观察项目上线后的反馈质量，当前支持：

- 查看全部反馈总览
- 分别查看每个模型的表现
- 统计每个模型的反馈数量、正确数、错误数
- 查看反馈准确率与平均置信度
- 查看最近错误样本，辅助后续模型优化

这部分很适合继续扩展为：

- 推理历史统计
- 模型耗时趋势
- 高频误识别类别分析
- 错误样本回流训练
